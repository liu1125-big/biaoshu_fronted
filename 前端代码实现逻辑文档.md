## 一、路由与导航体系

### 1.1 路由定义 (`App.tsx`)

```tsx
<Routes>
  <Route path="/" element={<Navigate to="/technical-plan" replace />} />
  <Route path="/technical-plan" element={<TechnicalPlanHome workflowKind="technical-plan" />} />
  <Route path="/existing-plan-expansion" element={<TechnicalPlanHome workflowKind="existing-plan-expansion" />} />
  <Route path="/document-knowledge-base" element={<KnowledgeBasePage />} />
</Routes>
```

- `/` 和 `/bid-generation` 都重定向到 `/technical-plan`
- **同一个组件 `TechnicalPlanHome`** 通过 `workflowKind` prop 区分两种模式：
  - `"technical-plan"` → 全新标书生成
  - `"existing-plan-expansion"` → 已有方案扩写（多一个"原方案"上传步骤）
- 知识库有独立页面组件

### 1.2 侧边栏菜单 (`menuConfig.ts` → `Sidebar.tsx`)

菜单数据结构为树形：
```typescript
AppMenuItem[] = [
  { id, label, description, children?: AppSubMenuItem[] }
]
```

`Sidebar.tsx` 循环渲染菜单项：
- 点击菜单项时，取 `item.children[0].id` 作为路由路径进行 `navigate()`
- 当前激活项通过 `location.pathname` 匹配
- 折叠模式下使用 Radix UI Tooltip 显示完整名称

## 二、API 通信层架构

这是接入后端时最关键的层次。所有前端与后端的通信都通过两层抽象完成。

### 2.1 端点常量 (`shared/api/endpoints.ts`)

```typescript
const BASE = import.meta.env.VITE_API_BASE || '';  // 读取 .env 中的后端地址

export const ENDPOINTS = {
  // 配置
  CONFIG_LOAD: `${BASE}/api/config`,
  CONFIG_SAVE: `${BASE}/api/config`,

  // AI
  AI_CHAT: `${BASE}/api/ai/chat`,
  AI_REQUEST_JSON: `${BASE}/api/ai/request-json`,

  // 文件解析
  FILE_PARSE: `${BASE}/api/file/parse`,

  // 标书生成 - 约 16 个端点
  TECHNICAL_PLAN_LOAD_STATE: `${BASE}/api/technical-plan/state`,
  TECHNICAL_PLAN_IMPORT_TENDER: `${BASE}/api/technical-plan/import-tender`,
  // ... 其他端点

  // 任务（SSE 实时推送）
  TASKS_START_BID_ANALYSIS: `${BASE}/api/tasks/bid-analysis`,
  TASKS_EVENTS: `${BASE}/api/tasks/events`,

  // 导出
  EXPORT_WORD: `${BASE}/api/export/word`,

  // 知识库 - 16 个端点
  KNOWLEDGE_BASE_LIST: `${BASE}/api/knowledge-base`,
  KNOWLEDGE_BASE_CREATE_FOLDER: `${BASE}/api/knowledge-base/folders`,
  KNOWLEDGE_BASE_UPLOAD_DOCUMENTS: `${BASE}/api/knowledge-base/documents/upload`,
  KNOWLEDGE_BASE_EVENTS: `${BASE}/api/knowledge-base/events`,
  // ... 其他端点
} as const;
```

**关键点：**
- `BASE` 来自环境变量 `VITE_API_BASE`，默认为空字符串
- 如果后端在同一域名下，不需要设置 `VITE_API_BASE`
- 如果后端在不同端口/域名，在 `.env` 中设置 `VITE_API_BASE=http://localhost:8080`
- 所有端点用 `as const` 标记为字面量类型，保证类型安全

### 2.2 API 客户端 (`shared/api/apiClient.ts`)

创建 axios 实例：
```typescript
const http = axios.create({ timeout: 300000 });  // 5 分钟超时
```

以对象字面量组织所有 API 方法，按业务模块分组：

```typescript
export const apiClient = {
  config: { load(), save() },
  ai: { chat(), requestJson() },
  file: { parse() },
  technicalPlan: {
    loadState(),           // GET    /api/technical-plan/state
    importTenderDocument(),// POST   /api/technical-plan/import-tender (FormData)
    readTenderMarkdown(),  // GET    /api/technical-plan/tender-markdown
    saveOutline(),         // PUT    /api/technical-plan/outline
    // ... 约 15 个方法
  },
  knowledgeBase: {
    list(),                // GET    /api/knowledge-base
    createFolder(),        // POST   /api/knowledge-base/folders
    uploadDocuments(),     // POST   /api/knowledge-base/documents/upload (FormData)
    readMarkdown(),        // GET    /api/knowledge-base/markdown?documentId=X
    onEvent(),             // SSE    /api/knowledge-base/events → EventSource
    // ... 16 个方法
  },
  tasks: {
    startBidAnalysis(),    // POST   /api/tasks/bid-analysis
    onTaskEvent(),         // SSE    /api/tasks/events → EventSource
    // ... 其他任务方法
  },
  export: {
    exportWord(),          // POST   /api/export/word
    onWordExportProgress(),// SSE    /api/export/word/progress → EventSource
  },
};
```

**HTTP 方法模式：**

| 操作 | HTTP 方法 | axios 写法 |
|------|-----------|------------|
| 查询 | GET | `http.get(url, { params })` |
| 创建 | POST | `http.post(url, body)` |
| 更新 | PUT | `http.put(url, body)` |
| 删除 | DELETE | `http.delete(url, { data: body })` |
| 文件上传 | POST | `http.post(url, formData)` — 使用 `FormData` |

**SSE 模式（用于实时推送）：**
```typescript
onEvent: (callback) => {
  const eventSource = new EventSource(ENDPOINTS.XXX_EVENTS);
  eventSource.onmessage = (msg) => {
    try { callback(JSON.parse(msg.data)); } catch {}
  };
  eventSource.onerror = () => eventSource.close();
  return () => eventSource.close();  // 返回取消订阅函数
}
```

### 2.3 接入后端时的修改指南

**场景 A：后端 API 路径与预期一致**
- 只需在 `.env` 中设置 `VITE_API_BASE=http://your-backend:port`
- **不需要修改任何代码**

**场景 B：后端 API 路径不同**
- 修改 `shared/api/endpoints.ts` 中对应的端点字符串
- 例如后端用 `/api/v1/kb/list` 而不是 `/api/knowledge-base`

**场景 C：请求/响应格式不同**
- 修改 `shared/api/apiClient.ts` 中对应方法的请求体或返回值解构
- 例如后端返回 `{ code: 0, data: { folders, documents } }` 而不是直接返回 `{ folders, documents }`

**场景 D：后端不需要某个功能**
- 将 `apiClient.ts` 中对应方法改为返回空数据（stub），例如：
  ```typescript
  list: async () => ({ folders: [], documents: [] }),
  ```

## 三、状态管理与数据流

本项目未使用 Redux/Zustand 等状态管理库。状态管理模式为：

### 3.1 状态提升（Lifting State Up）

每个 feature 的主页面组件（如 `TechnicalPlanHome`、`KnowledgeBasePage`）持有核心状态，通过 props 向下传递。

```
TechnicalPlanHome (state owner)
  ├─ DocumentAnalysisPage  ← props: tenderMarkdown, onFileImported, ...
  ├─ BidAnalysisPage       ← props: bidAnalysisConfig, onSave, ...
  ├─ OutlineEditPage       ← props: outline, onSave, ...
  ├─ GlobalFactsPage       ← props: globalFacts, onSave, ...
  └─ ContentEditPage       ← props: chapters, onSaveChapter, ...
```

### 3.2 自定义 Hook 模式

复杂的状态逻辑提取到自定义 Hook 中：

- **`useKnowledgeBase`** — 知识库核心状态：索引加载、文件夹 CRUD、文档 CRUD、SSE 事件、自动匹配
- **`useKnowledgeBaseViewer`** — 文档查看状态：三种查看模式、Markdown/条目/分析数据加载
- **`useKnowledgeBaseDrag`** — 拖拽排序：文件夹/文档的拖拽状态和操作
- **`useTechnicalPlanWorkflow`** — 标书工作流：步骤切换、前进/后退

Hook 返回状态 + 操作方法，页面组件解构使用：
```typescript
const kb = useKnowledgeBase({ showToast, showDocumentParseNotice });
// kb.index, kb.activeFolder, kb.createFolder(), kb.uploadDocuments(), ...
```

### 3.3 后端状态同步

有两种同步机制：

1. **请求-响应**（普通 HTTP）：
   ```
   用户操作 → 调用 apiClient 方法 → axios 请求 → 后端处理 → 返回数据 → 更新 React state → 重新渲染
   ```

2. **SSE 实时推送**（任务进度）：
   ```
   用户启动任务 → apiClient 发起 → 后端处理中 → EventSource 推送事件 → 回调更新 state → 实时更新 UI
   ```

## 四、标书生成（technical-plan）完整流程

### 4.1 工作流概览

技术方案生成分 5 个步骤，顺序执行：

```
STEP 01 → STEP 02 → STEP 03 → STEP 04 → STEP 05
 上传       AI分析     大纲编辑    全局事实    章节生成
 招标文件   标书理解    结构调整    企业信息    逐章AI写
```

每个步骤对应一个 Page 组件，由 `TechnicalPlanHome` 根据 `state.step` 条件渲染。

### 4.2 TechnicalPlanHome（主控制器，~1027行）

**核心职责：**
- 管理全局 `TechnicalPlanState`（步骤、文件、配置等）
- 检测 demo 模式（URL 参数 `?demo=1` + localStorage）
- 根据 `state.step` 条件渲染对应的步骤页面
- 通过 `useEffect` 在组件挂载/状态变化时与后端同步数据

**关键 state 字段：**
```typescript
interface TechnicalPlanState {
  step: TechnicalPlanStep;           // 当前步骤
  workflowKind: 'technical-plan' | 'existing-plan-expansion';
  tenderFile: TechnicalPlanTenderFile | null;     // 招标文件信息
  originalPlanFile: TechnicalPlanOriginalPlanFile | null; // 原方案信息
  bidAnalysisConfig: BidAnalysisConfig | null;    // 标书分析结果配置
  outlineConfig: OutlineConfig | null;            // 大纲配置
  outline: OutlineNode[];                          // 大纲数据
  globalFacts: GlobalFacts | null;                // 企业全局事实
  contentOptions: ContentOptions | null;           // 内容生成选项
  chapters: Record<string, ChapterContent>;        // 各章节内容
}
```

**demo 模式处理：**
- 检测方式：URL 参数 `?demo=1` 或 localStorage `demo-mode`
- 核心逻辑：`DocumentAnalysisPage` 在 demo 模式下使用浏览器的 `FileReader` 直接读取文件内容，跳过文件上传 API
- `TechnicalPlanHome` 中 markdown 加载 useEffect 在 demo 模式下跳过 API 调用，避免 SPA fallback 覆盖正确内容

### 4.3 步骤 01：文档分析 (`DocumentAnalysisPage`)

**功能：** 上传招标文件（和可选的原方案），展示解析后的 Markdown 内容。

**上传流程：**
```
用户点击"上传" → <input type="file"> 弹出文件选择
→ 选择文件 → input.onchange 触发
→ 创建 FormData → 添加 file
→ apiClient.technicalPlan.importTenderDocument(formData)
→ 后端解析文件为 Markdown
→ 返回 { state, markdown, needsSectionSelection? }
→ 如果有多个标段 → 弹出 BidSectionSelectorDialog
→ 否则直接设置 tenderMarkdown 并更新 state
```

**demo 模式 fallback：**
```typescript
// 当 API 调用失败且 demoMode 为 true 时：
if (demoMode && file) {
  const markdown = await readFileAsMarkdown(file);
  // FileReader 读取 .txt/.md 内容
  // 对 .docx/.pdf 显示占位提示
  onFileImported({ tenderFile: {...} }, markdown);
}
```

### 4.4 步骤 02：AI 标书理解 (`BidAnalysisPage`)

**功能：** 启动 AI 分析任务，提取评分项和需求要点。

**流程：**
```
用户配置分析参数 → 点击"开始分析"
→ apiClient.tasks.startBidAnalysis(config)
→ 后端开始处理
→ 同时建立 SSE 连接：apiClient.tasks.onTaskEvent(callback)
→ 实时接收进度事件（TaskEvent）
→ 更新分析结果展示
```

**SSE 事件处理模式：**
```typescript
useEffect(() => {
  const unsubscribe = apiClient.tasks.onTaskEvent((event) => {
    // event 包含进度、状态、结果等
    setTaskState(event.state);
    if (event.status === 'completed') {
      // 分析完成，展示结果
    }
  });
  return unsubscribe;  // 组件卸载时断开 SSE
}, []);
```

### 4.5 步骤 03：大纲编辑 (`OutlineEditPage`)

**功能：** AI 生成大纲后，支持手动拖拽调整层级和顺序。

**核心交互：**
- 拖拽排序：调整章节顺序
- 层级调整：升降级章节
- 增删改：手动添加/删除/编辑章节
- AI 重生成：指定某章节重新生成

### 4.6 步骤 04：全局事实 (`GlobalFactsPage`)

**功能：** 填写企业资质、业绩等全局信息。

这些信息会在步骤 05 中注入到每个章节的 AI 生成 prompt 中，确保全书一致性。

### 4.7 步骤 05：章节内容生成 (`ContentEditPage`)

**功能：** 逐章或批量使用 AI 生成内容，支持中止和重试。

**关键特性：**
- 并发控制（后台任务队列）
- 中止机制（调用 `pauseContentGeneration`）
- 手动编辑每章内容
- 实时进度显示

## 五、知识库（knowledge-base）完整流程

### 5.1 架构设计

```
KnowledgeBasePage (页面组装)
  ├─ useKnowledgeBase        → 核心状态管理（数据加载、CRUD、SSE）
  ├─ useKnowledgeBaseViewer  → 文档查看器状态（三种模式切换）
  ├─ useKnowledgeBaseDrag    → 拖拽排序（文件夹/文档）
  │
  ├─ KnowledgeDocumentViewer → 查看器组件（组合子组件）
  │   ├─ KnowledgeAnalysisView       → 分析调试视图
  │   ├─ KnowledgeItemCard           → 知识条目卡片
  │   ├─ KnowledgeItemSourceDialog   → 条目原文弹窗
  │   └─ DebuggableMarkdownContent   → 性能追踪容器
  │
  └─ KnowledgeMigrationDialog → 数据迁移弹窗
```

### 5.2 数据加载流程

```
组件挂载
  → useKnowledgeBase.useEffect
    → loadInitialData()
      → GET /api/knowledge-base
      → 验证响应格式（防止 Vite SPA fallback 返回 HTML）
      → setIndex(data) / setActiveFolderId
  → 同时建立 SSE:
    → new EventSource(/api/knowledge-base/events)
    → 当文档状态更新时 → setIndex + syncDocument
```

### 5.3 三种查看模式

| 模式 | 触发条件 | API 调用 | 展示内容 |
|------|---------|---------|---------|
| `items` | 点击"查看条目" | `readItems(docId)` | 知识条目列表 |
| `markdown` | 点击"查看 Markdown" | `readMarkdown(docId)` | 原始 Markdown 渲染 |
| `analysis` | 点击"分析调试"（需 dev 模式） | `readAnalysis(docId)` | AI 分析报告和候选条目 |

### 5.4 拖拽排序

支持两种拖拽：
1. **文件夹拖拽**：在侧边栏内排序文件夹
2. **文档拖拽**：在文档列表内排序 / 移动到不同文件夹

实现通过 HTML5 Drag and Drop API：
- `onDragStart` → 设置 `dragPayload` 和 `dataTransfer`
- `onDragOver` → 计算放置位置（上/下）+ 显示视觉指示
- `onDrop` → 调用 API（`reorderFolder` 或 `moveDocument`）→ 更新本地 state

### 5.5 防御性编程

由于后端可能未运行，所有数据加载处都加了验证：

```typescript
// 防止 Vite SPA fallback 返回 HTML 字符串污染状态
if (data && typeof data === 'object' && Array.isArray(data.folders) && Array.isArray(data.documents)) {
  setIndex(data);
}
// 否则保持空状态，页面正常渲染（显示"还没有文件夹"）
```

3. **其他功能**：不跳过 API 调用，由各组件自行处理错误（显示 error toast）

## 九、常见问题

### Q: 为什么点击知识库页面变白屏？
A: 后端未运行时，Vite 对 API 请求返回 `index.html`（HTML 内容）。如果代码未做防御性校验，HTML 字符串会被当作 JSON 对象存入 state，导致访问 `data.folders.length` 时抛出 TypeError。解决方案：所有 API 响应都加结构验证。

### Q: 为什么上传 MD 文件后显示的是 HTML 代码？
A: demo 模式下，`useEffect` 在 state 更新后调用 `readTenderMarkdown()` API，Vite SPA fallback 返回 `index.html` 覆盖了正确内容。解决方案：demo 模式下跳过 markdown 的 API 加载。

### Q: 如何新增一个功能模块？
A: 在 `src/features/` 下创建新目录，按现有模式组织（types.ts + components/ + hooks/ + pages/），在 `App.tsx` 添加路由，在 `menuConfig.ts` 添加菜单项。

### Q: 自定义 Hook 应该放在哪里？
A: feature 内专用的 Hook 放在 `features/<name>/hooks/` 下。跨 feature 共享的 Hook 放在 `shared/ui/` 下。
